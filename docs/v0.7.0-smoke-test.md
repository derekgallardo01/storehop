# v0.7.0 multi-user smoke test (Android, 2 physical devices)

Pre-Play-Closed-Testing manual validation for the Mike + Amanda
invite flow. Runs end-to-end against the production Firebase
project, so it doubles as a "Firestore rules are deployed
correctly" check.

## Prereqs

- **Two Android devices** signed in to two different Google accounts.
  Aliases: "Device A / Mike account" and "Device B / Amanda account".
- The v0.7.0 release APK installed on both (`./gradlew :app:bundleRelease`
  then sideload, OR pull the latest Play Closed Testing track).
- `firestore.rules` from the repo root **deployed in the Firebase
  Console** (Firestore → Rules → paste + publish). The flow will fail
  silently with PERMISSION_DENIED if the v0.6.x default rules are
  still live.
- Both devices on Wi-Fi or LTE; airplane mode breaks the invite-doc
  round-trip.

## Reset

Clear state on both devices before running:

```
adb -s <serial-A> shell pm clear com.storehop.app
adb -s <serial-B> shell pm clear com.storehop.app
```

`pm clear` wipes the local Room DB, the Firebase Auth credential
cache, and the per-app locale preference. Each device launches like
a first install on next open.

## Run

### 1. Migration smoke (v0.6.x → v0.7.0 in place)

Skip if both devices started fresh. Otherwise:

1. On Device A, install v0.6.10 first.
2. Add 3 items (e.g. "Milk", "Eggs", "Bread"), 2 stores ("Lidl",
   "Aldi"), mark Milk purchased at Lidl. Sign in with Google.
3. Side-load the v0.7.0 APK (Settings → Apps → Install).
4. Open the app.

**Pass when:** every item / store / category / xref / purchase
record is still visible exactly as before. Settings → Household
shows "Just you" + your uid as the household id.

### 2. Invite happy path

1. **Device A** (Mike): Settings → Household → "Generate invite
   code". Bottom sheet shows an 8-character code. Note it down.
2. **Device B** (Amanda): Sign in with Google (different account
   from Mike). Settings → Household → "Join with code". Paste the
   code. Tap "Join".
3. After ~1-2 sec Amanda's Items tab repopulates with Mike's
   items.
4. On Device B add a new item ("Cheese", tagged at Lidl).
5. On Device A pull-to-refresh the Items tab.

**Pass when:** Cheese appears on Mike's device with the same brand
+ stores Amanda set.

### 3. Cross-store cascade extends to households

1. On Device B (Amanda): tap Lidl in the Shop tab. Tap Milk to
   mark it purchased.
2. On Device A (Mike): pull-to-refresh the Aldi shopping list.

**Pass when:** Milk shows struck-through on Aldi as well. (The
cascade is the explicit design intent of sharing — call this out to
the user if it feels surprising.)

### 4. Invite expired

1. **Device A**: Generate a new invite code.
2. In the Firebase Console, open Firestore → /invites → the
   freshly-written doc. Edit `expiresAt` to a past timestamp (e.g.
   `1000`).
3. **Device B**: Try to redeem the code.

**Pass when:** the form shows the inline error "This invite code
has expired." (or the localized equivalent for the device's
locale).

### 5. Invite already-used

1. Generate a code on A, redeem on B (now both share the household).
2. Open a third device or wipe + reset Device B's app data + sign
   in as a third Google account.
3. Try to redeem the same code.

**Pass when:** "This invite code has already been used."

### 6. Leave household

1. On Device B (Amanda, currently in Mike's household): Settings →
   Household → Leave household → confirm.
2. Amanda's local DB wipes; a fresh personal household is created
   (empty).
3. On Device A (Mike): pull-to-refresh.

**Pass when:** Mike's data is intact (the household survived
Amanda leaving). Amanda's app shows the empty-state list.

### 7. Sign-out + sign-back-in

1. On Device B: Settings → Account → Sign out.
2. The app drops to an anonymous local-only household. Add a test
   item ("XX") — anonymous data, never synced.
3. Sign in with the same Google account that previously joined
   Mike's household.

**Pass when:** Device B rejoins its **own personal** household
(the one created after the previous Leave). Mike's household is
NOT auto-rejoined — leaving was permanent until re-invited. The
anonymous "XX" item gets claimed onto the rejoined uid via the
existing orphan-uid claim path.

### 8. Statistics scope

1. With both devices in Mike's household (Amanda re-invited):
   each device buys 2-3 items.
2. On Device A: Settings → Statistics.
3. On Device B: Settings → Statistics.

**Pass when:** Mike sees only HIS purchases, Amanda sees only
hers — even though they share the household. Per the v0.7.0 design.

## Failure-mode checks

Quick adversarial pokes worth running once:

- Tap "Join" with an empty field → button is disabled (no error
  fired).
- Type a 5-character token + tap Join → "Codes are 8 characters."
- Type a 9-character token + tap Join → same error (client-side).
- Pull the network cable during `acceptInvite` → snackbar with a
  network error; the invite doc stays unaccepted (try again later
  and it works).
- Two devices try to redeem the same code simultaneously → exactly
  one succeeds; the other gets `AlreadyUsed` on retry.

## Ship gate

All 8 happy-path scenarios + the failure-mode checks pass on both
devices before promoting v0.7.0 from Play Closed Testing to Open
Testing.

If any scenario fails, capture the device serial + the timestamp
+ a `adb logcat | grep -i storehop` excerpt and file an issue
referencing this runbook line number.
